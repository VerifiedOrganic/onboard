# Interactive Map Mode

Generate a clickable HTML map through `onboard:render_map`, giving the user a navigable
mental model rather than a code reference. The target is a flow or architecture view they
can understand in under two minutes by clicking through nodes.

## Primary Tool Path

Use `onboard:render_map` with `format: "html"`.

- If the user names an output path, pass it as `output_path`.
- If they name a topic, pass it as `topic`.
- For an inferred architecture map, omit explicit `nodes` and `edges`; the tool derives a
  package-level map from the code graph.
- For a focused flow or custom concept map, pass explicit `nodes` and `edges` grounded in
  `recon`, `trace_flow`, `repo_map`, `deps`, `schema`, `routes`, or `stacks`.

Manual HTML generation is a fallback only when the MCP tool is unavailable.

## What Good Output Contains

- A clickable diagram generated by `render_map` (`format: "html"` is self-contained and
  does not need CDN assets).
- 5-12 high-signal nodes for most maps. If the scope needs more, split the map or narrow
  the topic.
- Detail per node: plain-English description and relevant file paths.
- Pan/zoom behavior supplied by the generated HTML.

## Quality Checklist

- Node labels are concepts a human would name, not raw file names.
- Edges represent real relationships from tools or source inspection. If an edge is
  inferred, say so.
- The map answers the user's actual question; an auth-flow request gets an auth-flow map,
  not a whole-app map.
- Every node cites at least one real file path when a file is known.

## When To Prefer This Mode

Use this mode when the user is visual, the codebase is large/interconnected, or the request
is specifically to "map" or "show" the system. For learning a single flow deeply with
back-and-forth, conversational mode is better; for a durable text reference, cached-guide
mode is better.
